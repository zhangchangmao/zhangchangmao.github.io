1. 产生背景：为什么要使用 Skills？

大模型现状：

  • 特定领域不专： 通用模型难以理解公司内部的深度业务逻辑 [1]。
  • 复杂任务困难： 面对需要多步骤、跨工具的长链路任务时力不从心 [1]。

解决的核心问题：

Skills 作为“桥梁”，能够将封装经验知识转化为 AI 可直接运行的流程，实现按需自动使用 [1]。

2. 基本原理：三位一体架构

Agent Skills 的运行逻辑是基于 **Agent + Skills + Computer** 的深度协作：

【图示：左侧为 Agent configuration (含核心 Prompt)，右侧为 Agent virtual machine，中间通过 "use computer" 指令连接文件系统中的 Skills 目录 [1]】

  • Agent 配置： 在 System Prompt 中装备技能列表（如 pdf, docx, xlsx）[1]。
  • 运行环境： 智能体通过 Bash、Python 或 Nodejs 在虚拟运行环境中读取并运行技能脚本 [1]。

3. Skill 的内部构造

一个标准的 Skill 并不是一段杂乱的代码，而是一个结构化的文件夹：

【图示：SKILL 文件夹拆解图。左侧为 SKILL.md (含技能流程、文件说明)，右侧为辅助文件夹：examples, templates, scripts [1]】

  • SKILL.md： 核心文档。定义了 `name` 和 `description`，告知 AI 何时调用该技能 [1]。
  • reference.md： 作为索引，帮助 AI 快速检索背景信息，支持渐进式加载 [1]。
  • Scripts： 实际运行的 Python 或脚本代码 [1]。

4. 核心区分：Skills vs. MCP

Skills 和 MCP (Model Context Protocol) 解决的是本质不同的问题：

【图示：左侧 Skills 标注为“知识共享、简单 MD 文件、无需服务器”；右侧 MCP 标注为“功能扩展、需要编码、外部集成” [1]】

核心观点： 两者是互补共生的。Skills 分享知识经验，MCP 扩展外部集成，共同实现完美协作 [1]。

5. 优缺点：硬币的两面

优势

  • 轻量灵活： 基于 Markdown，人人可创。
  • 渐进式加载： 极大节省 Token 消耗 [1]。

缺点与挑战

  • 上手门槛： 终端 CLI 环境对普通用户有“劝退感” [1]。
  • 路径不统一： 不同工具（如 Cursor, Claude Code）存放路径各异 [1]。
